## Проверка валидности AS-IS схемы
При анализе артефактов выявлено несколько несоответствий между схемой (Артефакт 1) и фактической работой системы.

| Что указано на схеме | Что происходит в действительности | Вывод |
| :------ | :------ | :------ |
| REST API в режиме Real-time между POS и SAP ERP| Ночной обмен XML-файлами по FTP (Артефакт 2)| ❌ Схема неактуальна |
| Онлайн-обновление остатков и программы лояльности | Данные обновляются только ночью | ❌ Не соответствует действительности |
| POS напрямую работает с SAP | Между ними используется файловый обмен| ❌ Требуется исправление |
| Локальная БД магазина | Подтверждается всеми артефактами| ✅ Верно |
| Прямая работа с оборудованием через драйверы | Подтверждается тикетом INC-8415| ✅ Верно |

Таким образом, AS-IS диаграмма требует корректировки: интеграцию POS с SAP необходимо показать как пакетный обмен через FTP/XML, а не как REST API в реальном времени.

## Hardware Abstraction Layer (HAL)

Для устранения зависимости Cloud POS от конкретных моделей оборудования вводится Store Edge Agent, выполняющий роль Hardware Abstraction Layer (HAL).
Его функции:
* единый API для Cloud POS;
* взаимодействие с фискальными регистраторами;
* взаимодействие с банковскими терминалами различных производителей (Ingenico, Verifone, Сбер, Альфа и др.);
* хранение локального кэша;
* выполнение офлайн-операций;
* синхронизация данных после восстановления связи.

При появлении нового типа терминала изменяется только HAL (или добавляется соответствующий адаптер), а Cloud POS продолжает работать без изменений.

 Это устраняет проблему, описанную в тикете INC-8415, где ранее требовалась доработка и пересборка кассового приложения.

##  Стратегия Offline Mode
При потере связи с облаком магазин продолжает работать в ограниченном режиме. Store Edge Agent использует локальный кэш (SQLite), содержащий актуальный каталог товаров, цены, настройки кассы и очередь неотправленных операций.

В офлайн-режиме доступны:
* продажа товаров за наличные;
* печать фискального чека;
* хранение чеков и операций в локальной очереди.

Функции, требующие обращения к центральным сервисам (например, начисление и списание бонусов программы лояльности или онлайн-авторизация банковских карт), временно недоступны либо выполняются по правилам деградации, что соответствует ограничениям, указанным в артефактах.

##  Синхронизация после восстановления связи

После восстановления интернет-соединения Synchronization Service автоматически обнаруживает доступность облака и начинает отправку накопленных данных в порядке их создания. Передача выполняется через API с использованием уникальных идентификаторов операций и механизма идемпотентности, что исключает повторную обработку чеков при повторной отправке.
После успешной загрузки облачные сервисы обновляют центральную базу данных, SAP ERP, остатки товаров и информацию программы лояльности. Локальная очередь очищается только после получения подтверждения о приёме данных, что обеспечивает целостность информации и исключает её потерю даже при повторных обрывах связи.

## Как предложенная архитектура решает проблемы кейса

| Проблема |Решение |
 :------ | :------ |
| Локальные серверы в магазинах | Полностью заменены облачными сервисами и управляемыми БД |
| Толстый клиент | Заменён на Cloud POS (Web/планшет) |
| Привязка к конкретным терминалам | Устранена с помощью Hardware Abstraction Layer и Payment Gateway|
| Высокий TCO | Исключены локальные серверы, лицензии Windows Server и SQL Server, сокращены выезды ИТ-инженеров|
| Долгое подключение новых магазинов | Централизованная облачная конфигурация и автоматизированный онбординг позволяют сократить срок до 5 дней |
|Работа при отсутствии интернета | Обеспечивается локальным кэшем, очередью операций и последующей синхронизацией с облаком |
