# Аудит

## 1. Бизнес-проблемы

Основные цели бизнеса:

* сократить TCO магазина на 40%;
* уменьшить срок подключения нового магазина с 2 недель до 5 дней;
* отказаться от локальной инфраструктуры;
* стандартизировать интеграцию платежных терминалов;
* снизить нагрузку на команды Store Ops и Data Integration.

## 2. Анализ текущей архитектуры (As-Is)

### Архитектура магазина
В каждом магазине находятся:
* толстый POS-клиент;
* локальная БД;
* локальный сервер;
* встроенные драйверы оборудования;
* прямая интеграция с SAP.

Фактически это распределённый монолит.

### Центральная система

В HQ работает:

* SAP ERP
* Promo Engine
* Smart Fulfillment

SAP является master-системой для товаров, цен и справочников.

## 3. Несоответствия в артефактах

### Самое важное противоречие.

Артефакт 1

На схеме указано:

Real-time stock & loyalty update через REST API.

Артефакт 2

На самом деле:

* инженер отправляет CSV по email;
* товаровед вручную сопоставляет товары;
* SAP ночью в 02:00 выгружает XML;
* XML по FTP рассылается магазинам.

Следовательно:

* REST API в реальном времени отсутствует.

Это типичный пример устаревшей архитектурной документации.

## 4. Основные архитектурные проблемы

1. Толстый клиент POS

Внутри находятся одновременно:
* бизнес-логика;
* работа с БД;
* драйверы оборудования;
* интеграция с банком;
* печать чеков.
Нарушение принципов разделения ответственности.

2. Локальная БД

Каждый магазин имеет собственную SQL Server/PostgreSQL.

Недостатки:
* лицензии;
* резервное копирование;
* обслуживание;
* обновления;

Выход из строя диска останавливает магазин.
Это подтверждает тикет INC-8433.

3. Жесткая интеграция оборудования

POS знает конкретный протокол терминала.

Следствие - каждый новый терминал требует:
* написания драйвера;
* пересборки приложения;
* нового релиза.
Подтверждается INC-8415.

4. Ручной SKU Matching

Процесс практически полностью ручной.

Пайплайн выглядит так:

CSV

↓

Email

↓

Excel

↓

VLOOKUP

↓

ручное исправление

↓

SAP

↓

FTP

↓

магазины

Это главный источник больших затрат.

5. Сильная связанность

POS зависит одновременно от:
* SAP
* Local DB
* оборудования
* драйверов
* банков

Любое изменение распространяется сразу на несколько компонентов.

6. Большой технический долг

Используются:
FTP
* COM-port
* USB Driver
* Email
* Excel
* VLOOKUP
* ручные операции

Все это признаки легаси.

## 5. Проблемы эксплуатации

###  Из тикетов видно три основных сценария.

### Интернет пропал
Последствия:
* чек пробить можно;
* карты лояльности не работают;
* безналичная оплата невозможна.

Причина:
централизованные сервисы недоступны.

### Новый терминал

Нельзя открыть магазин без разработки нового драйвера.

Это полностью противоречит масштабируемости.

### Поломка сервера

Выход из строя одного диска полностью останавливает магазин.

Single Point of Failure.

## 6. Причины высокой стоимости владения (TCO)

На каждый магазин приходится:
* Windows Server
* SQL Server
* локальный сервер
* резервное копирование
* обновления
* антивирус
* обслуживание
* выезды инженеров

При 500 магазинах расходы масштабируются линейно.

## 7. Узкие места процесса интеграции M&A

* Сбор данных

Инженер лично приезжает в магазин.

* Передача

Через Email.

* Сопоставление

Excel + VLOOKUP.

* Проверка

До 80% товаров вручную.
Это подтверждается интервью.

*Распространение

Только ночью.

## 8. Корневые причины проблем

Архитектурные
* распределённый монолит;
* отсутствие облачной архитектуры;
* отсутствие сервисной декомпозиции;
* сильная связанность компонентов.

Интеграционные
* FTP;
* Email;
* Excel;
* ручной обмен

Инфраструктурные
* сервер в каждом магазине;
* локальная БД;
* платные лицензии.

Организационные
* ручной SKU Matching;
* необходимость выезда инженеров;
* ручная поддержка оборудования.

## 9. Возможные направления целевой архитектуры (To-Be)

На основании целей бизнеса уже просматриваются основные архитектурные изменения:
*переход на Cloud POS (тонкий клиент);
* ликвидация локальных серверов;
* централизованная облачная БД;
* автоматизированный SKU Matching с использованием ML/AI и обязательной 
* верификацией спорных совпадений;
* Payment Gateway вместо прямых драйверов терминалов;
* API-интеграции вместо FTP и Email;
* облачные SaaS/PaaS-сервисы вместо локальной инфраструктуры;
* централизованное управление конфигурацией магазинов.

## Вывод

### Предлагаемая стратегия соответствует целям VitaStore:
* снижение TCO более чем на 25% за счёт отказа от локальных серверов, SQL Server и толстого POS-клиента;
* сокращение срока подключения нового магазина до 5 дней благодаря автоматизации процесса SKU Matching и централизованному управлению конфигурацией;
* уменьшение нагрузки на Team Store Ops за счёт перехода на Cloud POS, Payment Gateway и SaaS-сервисы;
* сохранение конкурентных преимуществ (Smart Fulfillment, Click & Collect, Promo Engine и SAP ERP) как собственных решений, развиваемых внутри компани
