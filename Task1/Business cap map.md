## 2. Функции, которые сейчас дублируются или дороги в поддержке

| Возможность | Проблема |
| ------ | ------ |
| POS | Толстый клиент, отдельное сопровождение каждого магазина |
| Локальная БД | Более 500 отдельных БД, резервное копирование и обслуживание |
| Локальные серверы | Высокие расходы на лицензии Windows Server и SQL Server |
| Программа лояльности | Локальные реализации и даже ведение в тетрадях |
| Прием оплаты | Жесткая зависимость от драйверов терминалов |
| Прием товара | Разные процессы в купленных сетях |
| Расчет премий | Выполняется локально разными способами |
| SKU Matching | До 80% сопоставления выполняется вручную |
| Управление ценами | Передача XML по FTP только ночью |
| Инкассация | Локальные процедуры без централизации |

## 4. Матрица Buy vs Build
Бизнес-возможность | Решение | Целевая модель | Обоснование |
| ------ | ------ | ------ | ------ |
|POS |Buy |SaaS |Стандартная функция, не создает конкурентного преимущества|
|Программа лояльности |Buy |SaaS| Большинство возможностей доступны в готовых облачных продуктах|
Payment Gateway |Buy |SaaS |Стандартная|
|POS |Buy |SaaS |Позволяет отказаться от собственных драйверов оборудования|
|Управление оборудованием |Buy |SaaS |Существуют готовые сервисы Device Management|
|Мониторинг магазинов |Buy |SaaS |Стандартный процесс эксплуатации|
|Корпоративная почта и обмен файлами |Buy |SaaS |Не является конкурентным преимуществом|
|Локальные БД |Migrate |PaaS |Перенос в управляемую облачную БД избавит от локальных серверов и лицензий|
|Хранилище документов |Migrare |Cloud Storage |Снижение стоимости хранения и обслуживания|
|Интеграционные сервисы |Migrate |PaaS |Использование API Gateway, Integration Platform и очередей сообщений|
|SKU Matching |Build |PaaS |Требует собственной логики с ML/Fuzzy Matching и обязательной проверкой оператором
|Smart Fulfillment |Build |PaaS |Ключевое конкурентное преимущество компании|
|Promo Engine |Build |PaaS |Собственная система управления акциями|
|Click & Collect |Build |PaaS |Основа омниканальной стратегии|
|SAP ERP |In-house |On-Premise |Критически важная корпоративная система, перенос нецелесообразен|
|Финансовый учет |In-house |On-Premise| Высокие требования к безопасности и интеграции с SAP|
Кадровый учет |In-house |On-Premise| Уже централизован в SAP ERP|

## 5. Итоговая стратегия Sourcing Strategy
Модель | Возможности |
| ------ | ------ |
|SaaS (Buy)|Cloud POS, Payment Gateway, Loyalty, Device Management, Monitoring, Service Desk, корпоративные сервисы
|PaaS/IaaS (Migrate/Build in Cloud)|Облачные базы данных, API Gateway, Integration Platform, SKU Matching, Smart Fulfillment, Promo Engine, Click & Collect|
|In-house / On-Premise| SAP ERP, финансовый учет, кадровый учет, мастер-данные предприятия

## Вывод

### Предлагаемая стратегия соответствует целям VitaStore:
* снижение TCO более чем на 40% за счёт отказа от локальных серверов, SQL Server и толстого POS-клиента;
* сокращение срока подключения нового магазина до 5 дней благодаря автоматизации процесса SKU Matching и централизованному управлению конфигурацией;
* уменьшение нагрузки на Team Store Ops за счёт перехода на Cloud POS, Payment Gateway и SaaS-сервисы;
* сохранение конкурентных преимуществ (Smart Fulfillment, Click & Collect, Promo Engine и SAP ERP) как собственных решений, развиваемых внутри компании.
